HAVEN: Trust Accounting & Reservation Platform

A full stack trust accounting and reservation operations platform for property managers, with financial correctness enforced in the database itself.

Overview

HAVEN is a full stack trust accounting and reservation operations platform for property managers, built at HīTech Labs. It centralizes the daily accounting of a short and long term rental business, double entry AP and AR, bank and card reconciliation, platform payout matching, owner distributions, and month end close, and treats integrity as a first class constraint.

Trust balances cannot go negative, posted records cannot be silently edited, and every figure traces back to a journal entry and its supporting document.

Stack

  • Backend: Python 3.12, FastAPI, SQLAlchemy 2.0, Pydantic v2, Alembic, and PostgreSQL 16 (psycopg 3).
  • Frontend: React 19, TypeScript, Vite, TanStack Query, Table, and Virtual, dnd-kit, Radix UI, pdf.js, and Zod.
  • Infrastructure: Docker Compose, S3 compatible object storage (MinIO), and an outbox worker for integration events.

Architecture and engineering decisions

  • Ledger first accounting: every balance, statement, and report derives from double entry journal entries and subledgers rather than editable summary tables, so figures are always reconstructable and auditable.
  • Correctness enforced in the database: owner and trust subaccounts are blocked from going negative, and posted or finalized records are made immutable through PostgreSQL triggers, so corrections are reversals rather than edits.
  • SQL as the source of truth: the physical schema lives in 18 versioned raw SQL migrations run by Alembic, with check constraints, functional unique indexes, and immutability triggers enforced at the database layer, not just in application code.
  • Evidence first audit trail: invoices, receipts, ACH confirmations, bank statements, and generated packets are content addressed by SHA-256, stored in object storage, and linked directly to the records they support.
  • "Food Web" object graph: a navigation first data model where any record surfaces its related context, money trail, evidence, and downstream accounting impact without hunting across modules.
  • Integration first ingestion: booking platforms, bank and card feeds, and CSV sources map to canonical internal objects through a preview, validate, then commit import pipeline, sync jobs, and webhook events.
  • Layered, multi tenant API: a routers, services, and models backend exposing 200+ organization scoped REST endpoints, with role based access for Accountant, AP Clerk, Controller, and Owner, and a locked down owner portal.

Frontend

  • Reusable operations workbench: a single shell with a command palette, saved views, tabbed workspaces, and keyboard shortcuts hosts around 20 workspaces, including bill entry, bank and card inbox, payment batches, platform payout matching, cash application, reservation financials, trust distributions, month end close, and reporting and audit.
  • Edit in place workflows: virtualized work lists with inline, independently expandable review cards, so most actions happen without leaving the list.
  • Live document handling: in browser PDF viewing with pdf.js, drag and drop evidence attachment with dnd-kit, and file imports that preview and validate before anything is committed.
  • Server state architecture: TanStack Query drives caching and invalidation, with a mock and live data mode toggle that runs the UI against fixtures or the live API.
